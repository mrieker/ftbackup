ftbackup - Fault Tolerant Backup

    This program comes with ABSOLUTELY NO WARRANTY; for details,
    see LICENSE section below.

The ftbackup utility provides for restoring archived files even in the presence of errors in the archive media. This is achieved by recording the archive with redundancy blocks using an XOR scheme, and the integrity of each block of the archive is assured by a sequence number and a cryptographic hash of each block so we can be sure that no block is missing and the blocks are not corrupt.

The ftbackup utility has several commands:

  • backup : create an archive from an existing directory tree
  • compare : compare an existing archive to an existing directory tree
  • diff : compare two existing directory trees
  • help : display this file in web browser
  • history : look up files saved in database
  • license : print out the license info
  • list : list files saved in an archive
  • restore : restore files saved in an archive to a directory tree
  • version : print out the ftbackup version
  • xorvfy : verify the sequences, hashes and XOR blocks of an archive

ftbackup backup options saveset rootpath

  • options
    • -blocksize bs : sequence, hash, encrypt and write bs bytes at a time. Must be a power of two in range 4096 (4K) to 1073741824 (1G). Default is 32768 (32K).
    • -encrypt [:cipher] [:hash] key : encrypt the saveset.
      • cipher : AES Blowfish Camellia CAST128 CAST256 DES DES_EDE2 DES_EDE3 DES_XEX3 GOST IDEA MARS RC2 RC5 RC6 Rijndael SAFER_K SAFER_SK SEED Serpent SHARK SKIPJACK Square TEA Twofish XTEA, (AES and Rijndael are synonymous), default AES
      • hash : RIPEMD160 RIPEMD320 RIPEMD128 RIPEMD256 SHA1 SHA224 SHA256 SHA384 SHA512 Tiger Whirlpool, default SHA1
      • key : one of these forms:
        • - (single hyphen) : prompt at stdin
        • @filename : read from first line of file
        • else : the given literal string
      If no -encrypt option is given, an MD5 hash will be written to each block as a data integrity check.
    • -history [::histss] histdb : write saved filenames to a database.
      • histss - name of saveset written to database (defaults to saveset parameter value). This parameter does not have to identify an actual file, it is only used by the history command. Thus, it can be any string useful to identify the saveset.
      • histdb - database file. This file is created if it does not already exist, otherwise, the new files and saveset are added as needed.
    • -idirect : use O_DIRECT when reading files to be archived. Usually does not result in performance increase of the archiving itself, but avoids thrashing the cache with blocks that will be accessed only once.
    • -noxor : do not write any XOR redundancy blocks.
    • -odirect : use O_DIRECT when writing the saveset so as to avoid thrashing the cache with blocks of the saveset as they will be written only once and most likely not read for quite some time.
    • -osync : use O_SYNC when writing the saveset so as to see media write errors as the saveset is written. Implied by -odirect.
    • -record file : write the time the backup is started to the given file. This file can then be used by a -since option on a later backup command for doing incremental backups.
    • -segsize segsz : instead of writing the whole archive to a single file, break it up into a series of smaller files, each of size segsz bytes (the last one may be smaller). The files are named savesetnnnnnn where nnnnnn are sequential numbers starting at 000001.
      • The given segment size must be a multiple of the span size.
      • spansize = blocksize * (xorsc + 1) * xorgc
    • -since file : skip files with ctime earlier than given in file. Note that we use the date written within the file, not the date of the file itself. Default is to process all files.
    • -verbose : print name of each file just before it is processed.
    • -verbsec secs : print name of file just before it is processed, at secs intervals.
    • -xor xorsc,xorgc : specify how often to generate XOR redundancy blocks. xorsc says to write one XOR block per xorsc data blocks (default is 31). xorgc says to group xorgc XOR blocks together for writing (default is 2). Each is an integer in range 1 to 255.
  • saveset : name of archive to be written.
  • rootpath : root of tree to be archived. The entire contents of the tree will be archived except:
    • subdirectory contents currently being used as a mount point (the subdirectory is backed up as an empty directory)
    • other contents of a directory containing a file named ~SKIPDIR.FTB (the directory is backed up as a directory containing only the ~SKIPDIR.FTB file)
    • file listed in ~SKIPNAMES.FTB within a directory
    • files with ctime as recorded in -since file
Examples:
  • ftbackup backup home.ftb /home
    • Backs up the /home directory tree to a single saveset file home.ftb.
    • The blocksize is the default of 32768.
    • It is not encrypted.
    • O_DIRECT is not used for either reading the files nor writing the saveset, so the system disk cache will be used.
    • The default XOR blocks of one per 31 data blocks, written two at a time will be done.
  • ftbackup backup -blocksize 65536 -encrypt :idea @backup.key -history /var/log/root_disk.hist -idirect -odirect -record /var/log/root_disk.reco -segsize $((1024*1024*1024)) -verbsec 30 -xor 63,2 /sdd1/root.full.ftb/ /
    • Backs up the root filesystem (/) to a series of 1GB files (-segsize $((1024*1024*1024))) named /sdd1/root.full.ftb/000001, /sdd1/root.full.ftb/000002, /sdd1/root.full.ftb/000003, etc.
    • Files are read from the root filesystem using O_DIRECT (-idirect) so as not to fill up the cache with blocks from files that will probably not be read until the next full backup.
    • The saveset files are encrypted using IDEA and SHA1 with the key given in file backup.key.
    • Each filename as it is backed up is written to /var/log/root_disk.hist database
    • Record the date/time the backup was started in /var/log/root_disk.reco so that an incremental backup can be done later using -since /var/log/root_disk.reco.
    • The saveset is written in 64KB byte blocks (-blocksize 65536). 126 data blocks are written followed by 2 XOR blocks (-xor 63,2). Thus the span size is 64KB * 64 * 2 = 8MB, so there can't be more than one sequence of up to two consecutive corrupted blocks in every 8MB of the saveset.
    • The saveset is written using O_DIRECT (-odirect) so as not to fill up the cache with blocks that will probably not be read until a restore is needed.
    • The name of the file about to be processed is printed every 30 seconds (-verbsec 30).
~SKIPNAMES.FTB contents:
  • When creating a backup, ftbackup will skip files listed in any ~SKIPNAMES.FTB files encountered in the directory tree. Each line of ~SKIPNAMES.FTB is a name (possibly with a sub-directory relative to the directory of the ~SKIPNAMES.FTB file). The given name and/or sub-directory can contain wildcard characters.

ftbackup compare options saveset {savewildcard -to outputmapping} ...

  • options
    • -decrypt [:cipher] [:hash] key : as given to -encrypt when saveset written.
    • -incremental : when comparing a directory to a filesystem with existing contents in that directory, existing files not present in the directory being restored will elicit a compare error.
    • -simrderrs prob : simulate read error with probability of 1 out of prob blocks.
    • -verbose : print name of each file just before it is processed.
    • -verbsec secs : print name of file just before it is processed, at secs intervals.
  • saveset : name of archive to be compared.
  • savewildcard : select files from the saveset matching this wildcard. Use '**' to select all files.
  • -to : required keyword
  • outputmapping : where to compare files matching savewildcard to. outputmapping replaces the non-wildcard string on front of savewildcard.

ftbackup diff path1 path2

  • path1 : path of one directory to compare
  • path2 : path of other directory to compare to

ftbackup history options histdb [wildcard ...]

  • options
    • -ssbefore 'yyyy-mm-dd{ @T}hh:mm:ss' - only consider savesets created before the indicated UTC date/time
    • -sssince 'yyyy-mm-dd{ @T}hh:mm:ss' - only consider savesets created since the indicated UTC date/time
  • histdb - name of database file given to -history option on backup command line
  • wildcard - wildcard to search for, default is to list all files in the database

ftbackup history -delss options histdb wildcard ...

  • options
    • -ssbefore 'yyyy-mm-dd hh:mm:ss' - only consider savesets created before the indicated UTC date/time
    • -sssince 'yyyy-mm-dd hh:mm:ss' - only consider savesets created since the indicated UTC date/time
  • histdb - name of database file given to -history option on backup command line
  • wildcard - wildcard of savesets to delete

ftbackup history -listss options histdb [wildcard ...]

  • options
    • -ssbefore 'yyyy-mm-dd hh:mm:ss' - only consider savesets created before the indicated UTC date/time
    • -sssince 'yyyy-mm-dd hh:mm:ss' - only consider savesets created since the indicated UTC date/time
  • histdb - name of database file given to -history option on backup command line
  • wildcard - wildcard of savesets to list, default is to list all savesets in the database

ftbackup restore options saveset {savewildcard -to outputmapping} ...

  • options
    • -decrypt [:cipher] [:hash] key : as given to -encrypt when saveset written.
    • -incremental : when restoring a directory to an existing filesystem with existing contents in that directory, existing files not present in the directory being restored will be deleted.
    • -overwrite : overwrite an existing file with the file from the saveset. Otherwise, an error message is printed and the existing file remains as is.
    • -simrderrs prob : simulate read error with probability of 1 out of prob blocks.
    • -verbose : print name of each file just before it is processed.
    • -verbsec secs : print name of file just before it is processed, at secs intervals.
  • saveset : name of archive to be restored.
  • savewildcard : select files from the saveset matching this wildcard. Use '**' to select all files.
  • -to : required keyword
  • outputmapping : where to restore files matching savewildcard to. outputmapping replaces the non-wildcard string on front of savewildcard.
Examples:
  • ftbackup restore save.ftb '**' -to ''
    • Restore all files from save.ftb to where they came from.
  • ftbackup restore -decrypt :idea @backup.key -verbose root.full.ftb/ '/etc/**' -to /restored_etc/ '/var/**.log' -to /restored_var_log/
    • Restore from the encrypted root.full.ftb/ saveset (presumably segmented files in that directory named root.full.ftb/000001, root.full.ftb/000002, etc).
    • Print the name of each file just before restoring it (-verbose).
    • Restore files named beginning with, when saved, /etc/. Splice out the /etc/ and replace it with /restored_etc/, eg, /etc/hosts gets restored to /restored_etc/hosts.
    • Restore files named beginning with, when saved, /var/ and ending with .log. Replace the /var/ with /restored_var_log/, eg, /var/log/openvpn.log gets restored to /restored_var_log/log/openvpn.log.

ftbackup xorvfy options saveset

  • options
    • -decrypt [:cipher] [:hash] key : as given to -encrypt when saveset written.
  • saveset : name of archive to be verified.

Wildcards

  • \ - escapes for literal interpretation of following character
  • * - matches zero or more non-slash characters
  • ** - matches zero or more characters, including slashes
  • ? - matches any single character
  • [...] - matches a single character of name to at least one of the enclosed alternatives:
    • if first character of ... is a ! or a ^, the sense of the whole match is negated, ie, the name character must match none of the remaining ... characters
    • first character of remaining ... can be a ] to match the ] as a literal character
    • a single character - matches that single character
    • a single character, a hyphen, a single character - matches that range of characters
  • Use something like apostrophes (') to protect the wildcard from the shell attempting to expand it.
Examples:
  • *.tmp - match anything in current directory that ends in .tmp
  • **.tmp - match anything in any directory or sub-directory that end in .tmp
  • **[0-9A-Fa-f] - match anything in any directory or sub-directory that ends in an hexadecimal digit
  • **[!~#] - match anything in any directory or sub-directory that ends in neither ~ nor #

Saveset read errors

When encountering an I/O error while reading a saveset, such as for a restore, ftbackup will prompt the user at stdin for what action to take. The stdin must be a TTY. If not, the skip action will be taken.

Note: You can force skip mode without any prompting by redirecting stdin from something other than a TTY, such as /dev/null.

Available actions are:

  • abort - abort the restore by exiting
  • close - close the saveset file (so the saveset volume can be umounted if necessary)
  • retry - retry the read (after re-opening the saveset if necessary)
  • skip - skip reading this block and continue processing, recovering the skipped block via XORs if possible (after re-opening the saveset if necessary)
  • skipall - just as with skip, but assume skip action for all future read errors of the saveset.
  • control-Z - suspend restore and return to shell. The restore can be resumed by entering a fg shell command. This can be used to do things like umount then remount the saveset disk in case the filesystem has done something that can be remedied by such actions. If remounting, use the -r option if possible to remount the disk in read-only mode.

Saveset format

Savesets consists of a series of fixed-length blocks whose size was given on the backup command line. The final block is padded with 0xFF bytes as necessary to fill it.

There are data blocks and XOR blocks. The data blocks are numbered starting at 1. There will be xorgc * xorsc data blocks in a row followed by xorgc corresponding XOR blocks. The XOR blocks are also numbered starting at 1.

For example, if xorgc = 2 and xorsc = 31 there will be 62 data blocks followed by 2 XOR blocks, repeated on to the end of the saveset. Each set of 62 + 2 blocks is called a span. The final span of the saveset may be shorter than the other spans, but will include both XOR blocks. In this example with xorgc = 2, the odd-numbered data blocks are XORd together to compose the odd-numbered XOR block at the end of the span, and the even-numbered data blocks are XORd together to compose the even-numbered XOR block at the end of the span.

Each block, either data or XOR, has these fields:

  • 8-byte magic number consiting of the 8 characters ftbackup. Both data and XOR blocks have this same magic number.
  • 4-byte unsigned data block sequence number. Data blocks have their actual number. XOR blocks have the XOR of the corresponding data blocks sequence numbers.
  • 4-byte unsigned XOR block sequence number. Data blocks have zero in this field. XOR blocks have the XOR block number.
  • 16-byte encryption nonce. Will be zeroes if not encrypted. Will have zeroes at the beginning followed by 8 nonce bytes for 8-byte ciphers. This field is not XORd for the XOR blocks, but filled with its own random nonce.
  • content bytes filling most of the block. XOR blocks consist of the XOR of the corresponding data blocks.
  • 16,20,32-byte integrity hash. This field is not XORd for the XOR blocks, but filled with its own computed hash.

The hash field is the hash of all fields (including the nonce) except the hash field itself, just before encryption. Encryption covers the content bytes and hash fields only, thus allowing checking for magic number and sequence numbers without knowing the key. The integrity of the sequence numbers is assured as they are included in the encrypted hash.

The content field for data blocks consists of these sub-fields:

  • 4-byte byte offset, from very beginning of block, of first file header within the block or zero if no file header in the block.
  • 1-byte log2 block size (same in all blocks)
  • 1-byte xor block count (data blocks: zero; XOR blocks: number of data blocks XORd for this block)
  • 1-byte xor group count (same in all blocks)
  • 1-byte xor span count (same in all blocks)
  • data bytes to fill the field

The data bytes field alternates between file headers and file contents with no alignment or padding, overflowing from one data block to the next. File headers are never compressed, and only the contents of regular files and directories are compressed.
